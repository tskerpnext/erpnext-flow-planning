# 會計年度與關帳流程說明

適用地區：中國大陸
系統版本：ERPNext / Frappe v16
更新日期：2026-04-23

---

## 一、會計年度（Fiscal Year）

### 作用

會計年度是 ERPNext 所有財務活動的時間邊界，系統用它來：

| 作用 | 說明 |
|------|------|
| 限制過帳範圍 | 財務單據只能過帳到已開放的會計年度內 |
| 損益計算基準 | 收入、費用、利潤報表均以會計年度為統計單位 |
| 期初餘額歸零 | 損益科目在新年度執行結帳後自動歸零，重新累計 |
| 預算控制 | 預算（Budget）以會計年度為周期設定與追蹤 |
| 財務報表邊界 | 資產負債表、損益表均依會計年度產生 |

中國大陸標準會計年度為 **1月1日 ～ 12月31日**。

---

### 未建立新會計年度的影響

若新年度未建立，1月1日起所有涉及 GL 過帳的單據均無法提交：

```
❌ 系統報錯：
"Date 2027-01-01 is not in any active Fiscal Year"
```

| 操作 | 是否受影響 |
|------|----------|
| 建立單據（草稿） | ✅ 可以，草稿不過帳 |
| 提交財務單據 | ❌ 所有涉及 GL 的單據均擋住 |
| 銷售訂單 / 採購訂單 | ✅ 不過帳，不受影響 |
| 庫存單據（不涉及 GL） | ✅ 不受影響 |

**解決方式**：建立新會計年度即可立即恢復，建立完成後直接重試提交即可，不需重建單據。

> **建議**：每年 12 月中旬提前建立好下一年度，避免元旦當天財務人員無法開票。

---

### 建立新會計年度步驟

> **Accounting > Fiscal Year > 新建**

| 欄位 | 填寫範例 |
|------|---------|
| 名稱 | 2027 |
| 開始日期 | 2027-01-01 |
| 結束日期 | 2027-12-31 |
| 公司 | 選擇對應公司 |

建立後點「**設為預設（Set as Default）**」。

---

### 新舊年度同時開放的行為

```
2026 會計年度（未關帳）──┐
                         ├──▶ 兩個年度同時有效，單據均可提交
2027 會計年度（已建立）──┘
```

| 情況 | 結果 |
|------|------|
| 發票日期填 2026-xx-xx | 過帳到 2026 年度 |
| 發票日期填 2027-xx-xx | 過帳到 2027 年度 |
| 財務報表查 2026 | 仍可查，損益未結轉 |
| 財務報表查 2027 | 損益從零開始累計 |

## 二、關帳定義與期末結帳憑證的關係

### 關係說明

**關帳**是一個流程概念，包含多個步驟。
**期末結帳憑證（Period Closing Voucher）**只是關帳流程中的其中一個動作。

```
關帳流程（大）
    │
    ├─ 前置確認作業（各模組）
    ├─ 鎖帳（Accounting Period 關閉）
    ├─ 期末結帳憑證（Period Closing Voucher）← 只是這一步
    └─ 結帳後報表存檔
```

### ERPNext 關帳的兩個層次

| 層次 | 工具 | 作用 |
|------|------|------|
| 鎖帳 | Accounting Period | 關閉特定期間，禁止再對該期間過帳 |
| 結帳 | Period Closing Voucher | 損益科目餘額結轉留存收益，損益歸零 |

兩者可以獨立執行，系統均不強制要求。

---

### 不關帳的風險

| 風險 | 說明 |
|------|------|
| 損益未結轉 | 損益科目餘額不會進入留存收益，資產負債表不完整 |
| 舊帳可被異動 | 財務人員仍可補提舊年度單據，容易發生錯誤 |
| 報表不準確 | 未封帳的年度數字隨時可能被異動 |
| 稅務風險 | 帳面數字與增值稅申報數字可能對不上 |

---

## 三、關帳完整流程

### 涉及模組與順序

```
關帳前置作業
    │
    ├─【庫存模組】確認庫存評價與期末盤點
    ├─【固定資產模組】提列折舊
    ├─【薪資模組】確認薪資已入帳
    ├─【應收模組】確認發票與收款已核銷
    ├─【應付模組】確認採購發票與付款已核銷
    ├─【銀行模組】完成銀行對帳
    └─【稅務】增值稅申報數字確認
            │
            ▼
    【會計模組】鎖帳 → 結帳憑證 → 報表存檔
```

---

### 第一階段：前置確認

**庫存模組**
> Stock > Stock Reconciliation / Stock Ledger

- 確認期末實際盤點已完成
- 確認庫存評價方式（FIFO / 移動平均）計算正確
- 確認所有 Stock Entry 已提交，無草稿殘留

**固定資產模組**
> Accounting > Asset > Asset Depreciation

- 執行當期折舊（Depreciation）
- 確認折舊分錄已產生 GL Entry

**薪資模組**
> Payroll > Salary Slip / Payroll Entry

- 確認當期薪資單已提交
- 薪資 GL Entry 已過帳

**應收帳款**
> Accounting > Accounts Receivable

- 確認所有銷售發票已提交
- 確認收款已核銷，無異常未核銷餘額
- 對帳客戶帳齡報表

**應付帳款**
> Accounting > Accounts Payable

- 確認所有採購發票已提交
- 確認付款已核銷

**銀行對帳**
> Accounting > Bank Reconciliation

- 逐筆核對銀行流水與系統收付款憑證
- 確認餘額一致

**增值稅確認（中國大陸）**
> Accounting > General Ledger（篩選應交稅費科目）

- 核對應交增值稅科目餘額與申報數一致
- 若有差異，用日記帳憑證調整

---

### 第二階段：鎖帳（Accounting Period）

> **Accounting > Accounting Period > 選對應期間 > 關閉**

```
關閉後效果：
該期間 ──▶ 禁止新增 / 修改任何財務單據
           已提交單據不受影響
           若強行要過帳 ──▶ 系統報錯擋住
```

可設定鎖定的單據類型：Sales Invoice、Purchase Invoice、Journal Entry、Payment Entry、Stock Entry

---

### 第三階段：期末結帳憑證（Period Closing Voucher）

> **Accounting > Period Closing Voucher > 新建**

| 欄位 | 填寫說明 |
|------|---------|
| 期間結束日期 | 如 2026-12-31 |
| 結帳科目 | 選「留存收益」或「本年利潤」科目 |
| 備註 | 填寫年度說明 |

**執行後產生的分錄範例（淨利潤 800,000 元）：**

```
借：主營業務收入       1,000,000
借：其他收入              50,000
    貸：主營業務成本      200,000
    貸：管理費用           50,000
    貸：留存收益          800,000

結果：
損益科目全部歸零 ✅
留存收益增加 800,000 ✅
```

---

### 第四階段：結帳後報表存檔

> **Accounting > Financial Statements**

| 報表 | 說明 |
|------|------|
| 資產負債表（Balance Sheet） | 確認資產 = 負債 + 權益 |
| 損益表（Profit and Loss） | 確認收入、成本、利潤 |
| 總帳明細（General Ledger） | 備查所有分錄 |
| 帳齡分析（AR/AP Aging） | 確認應收應付無遺漏 |

建議匯出 PDF 存檔，作為年度財務憑證。

---

### 完整時序總覽

```
月底 / 年底
    │
    ├─ 確認庫存盤點完成
    ├─ 提列折舊
    ├─ 確認薪資入帳
    ├─ 應收應付核銷完畢
    ├─ 銀行對帳完成
    ├─ 增值稅申報數確認
    │
    ├─ 【鎖帳】Accounting Period 關閉
    ├─ 【結帳】Period Closing Voucher 執行
    └─ 【存檔】財務報表匯出備查
            │
            ▼
        新年度正式開帳
```

## 四、關帳對 MariaDB 資料的影響

### 結論

執行關帳與期末結帳憑證，MariaDB 的資料**完全不會移動、不會轉存、不會刪除**。

### 實際發生的事

Period Closing Voucher 執行後，MariaDB 只是新增記錄：

```
tabJournal Entry          ← 新增一筆結帳憑證
tabJournal Entry Account  ← 新增對應的借貸分錄明細
tabGL Entry               ← 新增對應的 GL 過帳記錄
```

舊年度所有資料原封不動留在原本的 TABLE 裡。

### 資料存放位置對照

| 資料類型 | MariaDB TABLE | 關帳後變化 |
|---------|--------------|-----------|
| 銷售發票 | `tabSales Invoice` | 不變 |
| GL 分錄 | `tabGL Entry` | 不變，只新增結帳那筆 |
| 收款憑證 | `tabPayment Entry` | 不變 |
| 付款台帳 | `tabPayment Ledger Entry` | 不變 |
| 結帳憑證 | `tabJournal Entry` | 新增一筆 |

### ERPNext 如何區分年度資料

不靠分開存放，靠日期欄位過濾：

```sql
-- 查詢 2026 年度的 GL Entry
SELECT * FROM `tabGL Entry`
WHERE posting_date BETWEEN '2026-01-01' AND '2026-12-31'
  AND company = 'SKT';
```

財務報表產生時，系統依 `posting_date` 篩選，不同年度的資料永遠共存在同一張 TABLE 中。

## 五、GL 分錄（GL Entry）說明

### GL 是什麼

**GL = General Ledger（總帳）**

GL 分錄就是總帳分錄，是每一筆財務交易在會計系統中留下的最底層記錄。每當提交一張財務單據，系統就會自動在 `tabGL Entry` 寫入記錄，記錄「哪個科目、借了多少、貸了多少」。

### 範例

```
提交一張銷售發票 100,000 元（含增值稅 13%）
            │
            ▼
系統自動產生 GL Entry：

科目                    借方        貸方
應收帳款             100,000
主營業務收入                       88,496
應交稅費－應交增值稅                11,504
```

### GL Entry 在 MariaDB 的主要欄位

| 欄位 | 說明 |
|------|------|
| `posting_date` | 過帳日期 |
| `account` | 會計科目 |
| `debit` | 借方金額 |
| `credit` | 貸方金額 |
| `voucher_type` | 來源單據類型（Sales Invoice / Payment Entry 等） |
| `voucher_no` | 來源單據編號 |
| `company` | 公司 |

### 哪些操作會產生 GL Entry

| 操作 | 產生 GL Entry |
|------|-------------|
| 銷售發票提交 | ✅ |
| 收款憑證提交 | ✅ |
| 採購發票提交 | ✅ |
| 日記帳憑證提交 | ✅ |
| 庫存入帳（含成本） | ✅ |
| 銷售訂單建立 | ❌ 不產生 |
| 採購訂單建立 | ❌ 不產生 |

### GL Entry 的重要性

> GL Entry 是 ERPNext 所有財務報表的資料來源。
> 損益表、資產負債表、總帳明細，全都是從 `tabGL Entry` 彙總計算出來的。

## 六、關鍵觀念整理

| 觀念 | ERPNext 實際情況 |
|------|----------------|
| 關帳是否強制 | 否，不關帳業務照常運作 |
| 未建新年度能否開票 | 否，提交時報錯 |
| 新舊年度可否同時開放 | 可以，依單據日期各自過帳 |
| 鎖帳後能否補帳 | 需先開啟 Accounting Period 才能補 |
| 期末結帳憑證能否反轉 | 可取消，損益科目餘額恢復 |
| 關帳後資料是否搬移 | 否，MariaDB 資料原地保留 |
| 報表如何區分年度 | 依 posting_date 篩選，非實體分離 |
