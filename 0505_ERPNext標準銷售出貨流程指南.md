# 0505_ERPNext標準銷售出貨流程指南 (權責分離模式)

> - **建立日期：** 2026-05-05
> - **最後更新：** 2026-05-05
> - **適用版本：** ERPNext v15+
> - **核心目標：** 建立一個權責分明、帳實相符、流程可追溯的標準銷售出貨流程，並解決相關權限配置問題。

---

## 一、 核心設計理念：為何要這樣規劃？

將「建立交貨單草稿」和「提交交貨單」這兩個動作，分給不同角色，是基於企業內部控制最核心的**「職責分離 (Separation of Duties)」**原則。其目的在於將**「商業操作」**與**「實物資產操作」**分開，以確保數據的準確性和安全性。

#### 1. 職責劃分

- **銷售用戶 (Sales User - 業務員):**
  - **核心職責：** 對**商業交易**負責。他們的工作是確認訂單、與客戶溝通、並發起發貨的「請求」。
  - **對應動作：** 建立「銷售訂單」，並從訂單生成「交貨單**草稿**」。這個草稿，本質上是一個給倉庫的「內部工作指令」。

- **倉庫用戶 (Stock User - 倉管員):**
  - **核心職責：** 對**實體庫存**負責。他們的工作是根據指令，準確地揀貨、打包，並記錄實物資產的轉移。
  - **對應動作：** 執行交貨單草稿上的任務，並在貨物**物理上離開倉庫**的那一刻，**「提交」**交貨單。這個「提交」動作，是對「實物已出庫」這一事實的最終確認，並觸發系統庫存的扣減。

#### 2. 為什麼倉庫用戶需要「地址」的讀取權限？

- **根本原因：** 倉庫用戶操作的核心單據是「交貨單」。這張單據為了完整地呈現一個發貨任務，必然會顯示「客戶名稱」和「送貨地址」等資訊。如果倉庫用戶沒有讀取「地址」這個文件類型的基本權限，系統在嘗試載入交貨單時，就會因為無法顯示完整的上下文資訊而報錯。
- **結論：** 授予「讀取地址」的權限，並非讓倉庫用戶去管理客戶主數據，而是為了讓他們能夠**順利地執行本職工作（處理交貨單）**所必需的「輔助權限」。

## 二、 角色與權限配置 (Role & Permission Setup)

要實現此流程，需要由**系統管理員 (System Manager)** 進行以下權限配置。

### 權限總覽表

| 角色 (Role) | 應有權限的文件 (DocType) | 權限細節 | 理由 |
| :--- | :--- | :--- | :--- |
| **銷售用戶** | `Sales Order` | 讀、寫、建立、提交 | 負責完成銷售訂單的生命週期。 |
| | `Delivery Note` | **僅「建立」** | 從銷售訂單發起發貨請求（建立草稿），但不能自己確認出庫。 |
| **倉庫用戶** | `Delivery Note` | 讀、寫、**提交** | 能夠查看並執行發貨任務，並在實物出庫後提交單據以扣減庫存。 |
| | `Sales Order` | **僅「讀取」** | 需要能看到待處理的銷售訂單，以便從中建立交貨單或安排工作。 |
| | `Address` | **僅「讀取」** | 為了在處理交貨單時能正常查看送貨地址，避免權限報錯。 |
| | `Serial No` / `Batch` | **僅「讀取」** | 為了在發貨時能選擇正確的序列號或批號。 |
| **倉庫經理** | `Delivery Note` | 讀、寫、提交、**取消**、**修改** | 除了倉庫用戶的權限外，額外擁有處理異常（如取消錯誤出貨）的更高權限。 |

### 權限配置詳細步驟

以下是如何為「倉庫用戶」這個角色，精確地補上他所缺少的必要權限的操作步驟。

1.  **進入角色權限管理器：**
    -   在 ERPNext 的全域搜尋列，輸入 `Role Permission Manager` 並進入。

2.  **選擇要修改的角色：**
    -   在 **Select Role** 這個欄位中，從下拉選單選擇 `Stock User` (倉庫用戶)。

3.  **為「地址 (Address)」新增讀取權限：**
    -   點擊頁面中間的 **「Add a New Rule」** 按鈕。
    -   在彈出的對話框中，**Select Document Type** 欄位，輸入並選擇 `Address`。
    -   在新增的 `Address` 這一行規則中，**只勾選「Read (讀取)」**，其他權限保持不勾選。

4.  **為「序列號 (Serial No)」新增讀取權限：**
    -   再次點擊 **「Add a New Rule」** 按鈕。
    -   在 **Select Document Type** 欄位，輸入並選擇 `Serial No`。
    -   在新增的 `Serial No` 這一行規則中，**只勾選「Read (讀取)」**。

5.  **為「批號 (Batch)」新增讀取權限：**
    -   重複以上步驟，為 `Batch` 文件類型也新增「Read」權限。

6.  **清除緩存並測試：**
    -   完成設定後，點擊頁面右上角的「清除緩存並刷新 (Clear Cache & Refresh)」。
    -   請倉庫用戶登出再重新登入，以確保新的權限設定已生效。

## 三、 標準作業流程 (SOP)

配置好權限後，系統中的標準出貨流程應如下操作：

1.  **業務接單 (由銷售用戶執行):**
    - 銷售用戶在系統中建立並**提交**一張「銷售訂單」。

2.  **發起出貨請求 (由銷售用戶執行):**
    - 銷售用戶打開剛提交的銷售訂單，點擊右上角的「建立」按鈕，選擇「交貨單」。
    - 系統會自動產生一張預填好所有資訊的交貨單**草稿**。
    - 銷售用戶核對無誤後，點擊**「儲存 (Save)」**。**（注意：不是提交）**

3.  **接收任務並揀貨 (由倉庫用戶執行):**
    - 倉庫用戶在他的「交貨單列表」中，透過篩選「狀態 = 草稿」，可以看到所有待處理的發貨任務。
    - 他根據交貨單草稿的內容，到倉庫中進行揀貨、打包。

4.  **確認出庫並提交 (由倉庫用戶執行):**
    - 貨物打包完成，交給物流公司，實體已離開倉庫。
    - 倉庫用戶回到電腦前，打開對應的交貨單草稿，最終確認所有資訊後，點擊**「提交 (Submit)」**按鈕。

5.  **系統自動化處理:**
    - 在倉庫用戶點擊「提交」的瞬間，系統會自動扣減庫存數量，並將銷售訂單的狀態更新為「部分/已交貨」。
    - 交貨單的狀態會變為「待開票」，等待財務部門進行後續的開票流程。

---

## 四、 總結

此流程透過將**「請求權」（建立草稿）**和**「執行與確認權」（提交單據）**分離，實現了銷售與倉儲部門之間的職責分工與相互制衡，是保障企業**權責分明、帳實相符、流程可追溯**的最佳實踐。
